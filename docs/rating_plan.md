# 마피아 레이팅 시행 계획

## 목표

마피아 게임의 레이팅은 단순 승률 순위가 아니라, 게임을 꾸준히 정상적으로 플레이한 사람의 실력을 장기적으로 보여주는 지표로 만든다.

기존 전적은 `stats.json`에 판수, 승패, 역할 횟수, 플레이 시간을 누적하고 있다. 레이팅도 같은 파일의 사용자별 전적에 필드를 추가해 관리한다.

## 기본 원칙

- 레이팅은 게임이 정상 종료된 뒤 한 번만 반영한다.
- 레이팅은 오를 수도 있고 깎일 수도 있다. 패배, 불리한 개인 행동, 무의미한 능력 사용은 점수 하락으로 반영한다.
- 중도 중지, 관리자 강제 중지, 오류로 정리된 게임은 반영하지 않는다.
- 팀 승패를 기본값으로 삼되, 특수직업별 행동 성과와 참가자 수 차이를 보정한다.
- 익명 모드여도 내부 계산은 Discord user id 기준으로 한다.
- 레이팅은 공개 순위용 점수이고, 기존 승률/판수/플레이시간 통계는 그대로 유지한다.

## 초기 규칙

- 초기 레이팅: `1000`
- 최저 표시 레이팅: `0`
- 배치 판수: `10판`
- 배치 중에는 레이팅을 저장하되 리더보드에서 `배치중`으로 표시한다.
- 리더보드 진입 조건: 정상 종료 게임 `10판 이상`
- 시즌 초기화는 바로 넣지 않고, 먼저 누적 레이팅으로 운영한다.

## 레이팅 계산 방식

기본은 Elo 방식을 사용한다.

```text
expected = 1 / (1 + 10 ** ((opponent_average - player_rating) / 400))
delta = k * (score - expected)
new_rating = old_rating + delta
```

마피아는 1대1이 아니라 팀 게임이므로 `opponent_average`는 상대 진영 생존/참가자의 평균 레이팅으로 계산한다.

## 레이팅 차감 규칙

패배하면 기본적으로 레이팅이 내려간다. 또 승리하더라도 개인 행동 보정이 크게 나쁘면 상승 폭이 줄어들 수 있다.

차감은 다음 상황에서 발생한다.

- 소속 진영 패배
- 본인 역할의 핵심 능력을 끝까지 사용하지 않음
- 능력 사용이 팀에 불리한 방향으로 작동함
- 특수직업이 접선, 조사, 치료, 포교, 청부 등 핵심 임무에 실패함
- 너무 유리한 평균 레이팅 팀으로 약한 팀을 상대로 패배함

단, 한 판에서 너무 크게 떨어지지 않도록 최종 변동 폭은 제한한다.

```text
final_delta = clamp(final_delta, -50, +50)
```

배치 중에는 변동 폭이 커질 수 있지만, 그래도 한 판 최대 하락은 `-60`을 넘기지 않는다.

## 승패 점수

| 결과 | score |
| --- | ---: |
| 소속 진영 승리 | 1.0 |
| 소속 진영 패배 | 0.0 |
| 조커 단독 승리 | 조커 1.0, 나머지 0.0 |
| 교주팀 승리 | 최종 교주팀 1.0, 나머지 0.0 |

도굴꾼처럼 게임 중 진영이 바뀌는 역할은 최종 승리 판정 함수 기준을 따른다.

## K값

K값은 레이팅 변동 폭이다.

| 조건 | K |
| --- | ---: |
| 10판 미만 | 40 |
| 10판 이상 30판 미만 | 32 |
| 30판 이상 | 24 |

판수가 적을 때는 빠르게 자리 잡고, 오래 플레이한 사람은 변동이 안정되게 한다.

## 팀 평균 계산

각 플레이어의 상대 평균은 다음 기준으로 잡는다.

- 시민팀 플레이어: 최종 마피아팀 평균 레이팅을 상대 평균으로 사용한다.
- 마피아팀 플레이어: 최종 시민팀 평균 레이팅을 상대 평균으로 사용한다.
- 교주팀 플레이어: 비교주팀 전체 평균 레이팅을 상대 평균으로 사용한다.
- 조커: 전체 비교커 플레이어 평균 레이팅을 상대 평균으로 사용한다.

상대 팀이 비어 있거나 계산이 불가능하면 전체 참가자 평균을 사용한다.

## 인원 보정

참가자 수에 따라 변동 폭을 조금 조정한다.

| 참가자 수 | 배율 |
| --- | ---: |
| 3명 이하 | 0.6 |
| 4명~6명 | 0.85 |
| 7명~10명 | 1.0 |
| 11명 이상 | 1.1 |

소규모 게임은 운 영향이 커서 변동을 줄인다.

## 개인 행동 보정

팀 승패만으로는 특수직업의 활약을 충분히 반영하기 어렵다. 그래서 팀 Elo 변동에 `role_delta`를 추가한다.

```text
team_delta = round(base_delta * player_count_multiplier)
role_delta = clamp(sum(role_impact_points), -10, +10)
final_delta = clamp(team_delta + role_delta, -50, +50)
```

원칙:

- 팀 승패가 가장 큰 비중을 가진다.
- 특수직업 보정은 한 판 최대 `+10/-10`까지만 반영한다.
- 패배팀도 좋은 플레이를 하면 하락 폭이 줄어들 수 있다.
- 승리팀도 핵심 역할을 방치하면 상승 폭이 줄어들 수 있다.
- 패배팀이 개인 보정만으로 큰 폭 상승하는 일은 막는다. 패배팀의 최종 상승은 최대 `+3`으로 제한한다.

## 역할 보정 운영 방식

초기 버전은 위 표의 고정 점수로 시작한다. 이후 실제 서버 데이터가 쌓이면 역할별 평균 승률과 평균 `role_delta`를 보고 조정한다.

조정 기준:

- 특정 역할의 평균 승률이 지나치게 높으면 좋은 행동 점수를 줄인다.
- 특정 역할의 평균 승률이 지나치게 낮으면 실패 감점을 줄인다.
- 개인 보정 때문에 팀 패배를 무시하는 플레이가 늘어나면 `role_delta` 상한을 낮춘다.
- 역할별 데이터가 30판 미만이면 자동 조정하지 않는다.

## 최종 변동 공식

```text
base_delta = k * (score - expected)
team_delta = round(base_delta * player_count_multiplier)
role_delta = clamp(sum(role_impact_points), -10, +10)
final_delta = clamp(team_delta + role_delta, -50, +50)
```

최종 레이팅은 다음처럼 제한한다.

```text
rating = max(0, rating + final_delta)
```

## 저장 구조

기존 `stats.json` 사용자 entry에 아래 필드를 추가한다.

```json
{
  "rating": 1000,
  "rating_games": 0,
  "rating_peak": 1000,
  "rating_history": [
    {
      "game_id": "2026-06-15T21:30:00+09:00",
      "before": 1000,
      "after": 1018,
      "delta": 18,
      "team_delta": 14,
      "role_delta": 4,
      "role": "경찰",
      "team": "시민팀",
      "winner": "시민",
      "players": 8,
      "rating_reasons": [
        "시민팀 승리",
        "마피아 판정 성공"
      ]
    }
  ]
}
```

`rating_history`는 최근 20개만 유지한다. 파일이 너무 커지는 것을 막기 위해 오래된 기록은 제거한다.

## 게임 기록 ID

게임 ID는 정상 종료 시점에 생성한다.

권장 형식:

```text
YYYY-MM-DDTHH:MM:SS+09:00-guild_id-channel_id
```

이 ID는 중복 반영 방지와 추후 감사 로그에 사용한다.

## 중복 반영 방지

`RunningGame`에 이미 `stats_recorded`가 있다. 레이팅도 같은 종료 처리에서 같이 반영한다.

추가로 `stats.json` 최상단에 최근 반영 게임 ID를 저장한다.

```json
{
  "rated_games": ["..."],
  "users": {}
}
```

`rated_games`는 최근 200개만 유지한다.

## 명령어 계획

### `/레이팅`

본인의 레이팅 정보를 보여준다.

표시 항목:

- 현재 레이팅
- 최고 레이팅
- 배치 여부
- 최근 변동
- 레이팅 반영 판수
- 기존 승률/판수 요약

### `/레이팅 @유저`

특정 유저의 레이팅 정보를 보여준다.

### `/레이팅순위`

레이팅 기준 리더보드를 보여준다.

기본 정렬:

1. 레이팅 높은 순
2. rating_games 높은 순
3. wins 높은 순
4. 이름 오름차순

배치 중인 사람은 기본 순위에서 제외하고, 옵션으로 볼 수 있게 한다.

### `/레이팅초기화`

관리자 전용 명령어.

옵션:

- 전체 초기화
- 특정 유저 초기화
- 레이팅만 초기화하고 기존 승패 전적은 유지

## 리더보드 표시 예시

```text
레이팅 순위
1. 홍길동 - 1242점 / 38판 / 최고 1280점
2. 김철수 - 1188점 / 21판 / 최고 1195점
3. 박영희 - 1150점 / 14판 / 최고 1150점

배치중
- 이민수 - 1031점 / 6판
```

## 내정보 표시 예시

기존 `/내정보`에 레이팅 줄을 추가한다.

```text
홍길동님의 전적
레이팅: 1242점 (최고 1280점)
레이팅 판수: 38판
전체 게임: 42판
승리/패배: 24승 18패
승률: 57.1%
마피아팀 플레이: 12회
게임시간: 330분
```

## 반영하지 않는 게임

아래 게임은 레이팅에 반영하지 않는다.

- 참가자 3명 미만
- 정상 승리 조건이 아니라 관리자 중지로 끝난 게임
- 봇 오류로 자동 정리된 게임
- 테스트 모드 게임
- 같은 유저가 여러 계정으로 참가했다고 관리자가 판단해 무효 처리한 게임

## 운영 정책

- 첫 도입 후 2주 동안은 “프리시즌”으로 운영한다.
- 프리시즌에는 레이팅을 계산하지만, 순위 보상이나 공식 순위로 쓰지 않는다.
- 프리시즌 종료 후 K값, 인원 배율, 배치 판수를 조정한다.
- 조정 후에도 기존 레이팅은 유지하고, 필요하면 관리자 명령어로 시즌 초기화를 한다.

## 구현 순서

1. `stats_store.py`에 rating 기본 필드 추가
2. 레이팅 계산 함수 추가
3. 역할별 `role_impact_points` 수집 함수 추가
4. `record_game_stats()` 종료 처리에서 승패 기록 후 레이팅 반영
5. 중복 반영 방지용 `rated_games` 저장
6. `/레이팅`, `/레이팅순위`, `/레이팅초기화` 명령어 추가
7. `/내정보`, `/리더보드`에 레이팅 기준 추가
8. `simulate_game.py`에 레이팅 필드 생성, 상승, 하락, 역할 보정 테스트 추가
9. 실제 서버에서 프리시즌으로 검증

## 테스트 계획

- 신규 유저가 첫 게임 후 `rating`, `rating_games`, `rating_peak` 필드를 갖는지 확인
- 승리팀은 레이팅이 오르고 패배팀은 내려가는지 확인
- 승리팀도 나쁜 역할 보정이 있으면 상승 폭이 줄어드는지 확인
- 패배팀도 좋은 역할 보정이 있으면 하락 폭이 줄어드는지 확인
- 강한 팀을 이겼을 때 더 많이 오르는지 확인
- 약한 팀에게 졌을 때 더 많이 떨어지는지 확인
- 한 판 최대 상승/하락 제한이 지켜지는지 확인
- 조커 단독 승리 시 조커만 승리 점수로 계산되는지 확인
- 교주팀 승리 시 최종 교주팀만 승리 점수로 계산되는지 확인
- 특수직업별 `rating_reasons`가 기록되는지 확인
- 관리자 중지 게임은 레이팅이 반영되지 않는지 확인
- 같은 게임이 두 번 기록되지 않는지 확인
- `/레이팅순위`에서 10판 미만 유저가 기본 순위에서 제외되는지 확인
